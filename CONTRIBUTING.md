# Contributing to EVM

Thank you for considering contributing to the EVM (Embedded Verification Module) framework!

## Code of Conduct

Be respectful, professional, and constructive in all interactions.

## How to Contribute

### Reporting Issues

- Use GitHub Issues to report bugs or request features
- Include clear descriptions and reproduction steps
- Provide relevant code snippets or examples

### Pull Requests

1. **Fork the repository**
   ```bash
   git clone https://github.com/DifferentialAudioInc/evm-sv.git
   ```

2. **Create a feature branch**
   ```bash
   git checkout -b feature/your-feature-name
   ```

3. **Make your changes**
   - Follow existing coding style
   - Add tests if applicable
   - Update documentation as needed

4. **Commit with clear messages**
   ```bash
   git commit -m "Add feature: description of what you added"
   ```

5. **Push and create pull request**
   ```bash
   git push origin feature/your-feature-name
   ```

## Coding Standards

### Python (CSR Generator)

- Follow PEP 8 style guide
- Use descriptive variable names
- Add docstrings to functions and classes
- Keep functions focused and modular

### SystemVerilog (Verification Framework)

- Follow existing naming conventions
- Use consistent indentation (4 spaces)
- Add comments for complex logic
- Keep classes focused on single responsibility

## Testing

- Test your changes locally before submitting
- For CSR generator: test with the example YAML file
- For verification framework: ensure changes don't break existing code

## Documentation

- Update README.md if adding new features
- Update tool-specific READMEs in subdirectories
- Add examples if introducing new functionality
